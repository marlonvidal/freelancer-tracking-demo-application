## Part 1 — Epic review

### What went well

- **Clear story sequencing:** 1.1 (route + shell) → 1.2 (nav) → 1.3 (persistence) matched dependencies and avoided scope bleed; each story’s “out of scope” sections stayed respected.
- **i18n and testing hygiene:** After 1.1 review feedback, the team consistently used `data-testid`, seeded locale in E2E where needed, and kept user-visible strings in `LanguageContext` (en + pt).
- **Architecture alignment:** `AppProvider` on `/earnings`, shared `Header` with optional Kanban props, dedicated persistence key separate from Kanban `AppState`, and `clearState()` clearing both — matches `docs/architecture.md` and PRD persistence intent.
- **Review → fix loop:** 1.1 and 1.3 addressed concrete patches (`document.title` lifecycle, finite date coercion, E2E seeding for AC4, nav `aria-label`). 1.2 code review improved a11y (landmark labeling) and tracked deferrals explicitly.

### Challenges and friction (systems, not individuals)

- **Cross-cutting conventions:** `@/` imports were called out in 1.1 and 1.2 reviews; `App.tsx` still mixes relative and alias imports — a recurring **process** gap (merge whenever that file is touched).
- **E2E stability vs. product behavior:** English-only assertions and wall-clock NFR checks were flagged as flake or false-confidence risks; mitigated where fixed, with one NFR item explicitly **deferred** as environmental.
- **Interim UI in 1.3:** Minimal `Select`-based controls are correct for the story but create a **handoff** dependency: Epic 4 must replace UI without breaking persistence API — already documented in story notes; worth tracking during Epic 4 kickoff.
