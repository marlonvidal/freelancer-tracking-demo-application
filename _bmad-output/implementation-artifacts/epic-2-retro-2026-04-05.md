## Part 1 — Epic Review

### What Went Well

- **FR26 formula parity achieved — and a latent bug fixed:** Story 2.1 discovered and corrected a `||` vs `??` operator mistake in `AppContext.getTaskRate`. Tasks with `hourlyRate: 0` were silently falling through to client rate instead of respecting the task-level zero override. The earnings engine work surfaced and fixed a real production bug that was invisible in the Kanban view.

- **Composable pure function architecture:** Story 2.2 composed entirely on Story 2.1 primitives — `filterTasksForEarnings`, `getTaskBillableRevenue`, `getEffectiveHourlyRate` — with no re-implementation. Epic 3 chart stories can use the same functions with zero calculation drift risk.

- **ATDD pipeline solidified as a habit:** Both stories had skipped Playwright ATDD specs, unskipped them, and shipped them green. 19 scenarios for Story 2.1, 16 for Story 2.2. The pattern is now proven and repeatable.

- **Code review caught real issues before merge:**
  - Story 2.1: Inclusive `endMs` boundary not asserted; duplicate-tag dedupe coverage missing (both patched)
  - Story 2.2: Hardcoded English `"total / billable"` in task count card caught and fixed with 2 new i18n keys before merge

- **`useMemo` dependency discipline:** `EarningsDashboard.tsx` memoizes on `[appState.tasks, appState.clients, state]` — not `Date.now()` on every render — preventing unnecessary recalculation cycles.

### Challenges and Friction (Systems, Not Individuals)

- **Playwright parallel-worker flake (Story 2.1):** Default parallel Playwright execution caused intermittent timeouts under load. Workaround (`--workers=1`) was discovered during implementation but not centrally documented. This will compound in Epic 3 with SVG chart rendering tests being timing-sensitive.

- **E2E substring collision (Story 2.2):** `getByText("Billable Revenue")` matched inside "Non-Billable Revenue" due to Playwright strict-mode. Fixed with `{ exact: true }`, but the pattern was not written down — it will be re-discovered without a standing convention.

- **Default app-state contamination in E2E (Story 2.2):** A zero-state test assumed an empty task list, but the app default ships 5 sample tasks. Required explicit empty `freelancer-kanban-data` seeding. Not documented as a standing rule.

- **Convention discipline lags documentation:** The team correctly follows established E2E patterns (locale seeding, `data-testid`, `{ exact: true }`) but these exist only in story dev notes, not in `project-context.md`. Each new story rediscovers them.

- **`getTotalRevenue` in `AppContext` is unfiltered:** The dashboard engine filters by date; the global Kanban `getTotalRevenue` does not. Currently low risk, but a potential user-visible inconsistency if Epic 5 renders summary metrics alongside Kanban totals.

## Part 2 — Next Epic Preparation (Epic 3: Chart Visualizations)

**Epic goal:** Implement interactive, responsive charts for Customer, Project, and Tag revenue breakdowns with tooltips, legends, and full accessibility support.

**Dependencies on Epic 2:**
- `calculateRevenueByCustomer` / `calculateRevenueByProject` / `calculateRevenueByTag` (Stories 3.1, 3.2, 3.3)
- Tag split semantics: `taskRevenue / tags.length` per tag; `Untagged` sentinel for untagged tasks (Story 3.3 critical)
- `resolveDateRangeMs` + `filterTasksForEarnings` remain the single source of truth
- `SummaryMetrics` / `calculateSummaryMetrics` surface already wired in `EarningsDashboard.tsx`

**New technical territory in Epic 3:**
- recharts `PieChart` / `ResponsiveContainer` — not yet tested in this codebase at scale
- SVG DOM assertions in Playwright — different patterns from regular HTML assertions
- Tooltip hover interactions and legend clicks — recharts internal state management
- NFR-P1: 2-second render budget with up to 5,000 tasks — performance unknown at this scale

**Preparation concerns surfaced:**
1. Recharts SVG testing patterns are undefined — Playwright queries differ significantly for SVG `<text>` elements vs HTML text
2. `--workers=1` convention undocumented — chart rendering tests will be more timing-sensitive than pure function tests
3. Tag split display contract (`1/N` revenue per tag) must be explicit in Story 3.3 spec to avoid developer re-deriving the math incorrectly
4. Performance baseline unknown — NFR-P1 should be on the radar from Story 3.1, not just Story 3.4
